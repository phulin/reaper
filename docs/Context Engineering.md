# Shared state for multi-agent binary reverse engineering

**A multi-agent RE system should center on a PostgreSQL + pgvector knowledge base implementing the blackboard architecture pattern, with hierarchical summarization providing progressive context disclosure to fresh agents.** This design balances structured metadata (function signatures, call graphs) with semantic search over unstructured analysis notes, supports concurrent writes from multiple agents with ACID guarantees, and draws on a rapidly maturing ecosystem of LLM-RE tools and agent frameworks. The field has moved fast since 2024: specialized decompilation LLMs now rival GPT-4o, MCP servers exist for every major RE tool, and at least one production multi-agent RE pipeline (SentinelOne's adversarial consensus engine) has been deployed. But no off-the-shelf system solves the full shared-state problem — you'll need to compose one from existing pieces.

## The knowledge base architecture that actually works

The core challenge is dual-natured: agents need **structured queries** (call graph traversal, cross-reference lookups, type resolution) alongside **semantic search** (finding functions behaviorally similar to a known pattern, retrieving relevant prior analysis notes). The most practical architecture uses **PostgreSQL as a single unified store** combining relational tables, pgvector embeddings, full-text search, and optionally Apache AGE for graph queries — all in one ACID-compliant database.

The schema should capture the natural hierarchy of binary analysis: **binary → modules → functions → basic blocks**, with a polymorphic annotation system that lets any agent attach structured findings at any level. A concrete schema includes:

- **`functions`** table with columns for address, proposed name, original symbol name, reconstructed signature, calling convention, decompiled pseudocode, AI-generated summary, complexity score, tags, and critically both a **code embedding** (`vector(768)`) and a **summary embedding** (`vector(384)`) for semantic search
- **`annotations`** table using a polymorphic target (type + ID) so agents can annotate functions, blocks, addresses, or the whole binary, with `agent_id`, `confidence` (0.0–1.0), `version`, and a `supersedes` self-referential FK for version chains
- **`call_graph_edges`** and **`cross_references`** tables for structural queries, with indexes supporting common traversal patterns
- **`modules`** table for higher-level groupings, representing logical components (crypto subsystem, network layer, command dispatcher) that agents identify during analysis
- **`data_types`** table storing reconstructed structs, enums, and typedefs as JSONB definitions with provenance tracking

This single-database approach eliminates the synchronization headache of maintaining consistency across separate graph and vector stores. pgvector's HNSW indexes deliver sub-millisecond approximate nearest neighbor search, while standard SQL handles structural queries. For teams needing serious graph traversal (10+ hop reachability queries, community detection for automatic module identification), **Neo4j as a secondary graph tier** synchronized from PostgreSQL is justified — but for most binaries, PostgreSQL with recursive CTEs or Apache AGE handles call graph queries adequately.

**Hybrid search is essential.** Agents need to combine structural filters with semantic similarity — for example, "find functions called by `dispatch_command` that are semantically similar to known encryption routines." Reciprocal Rank Fusion (RRF) combining BM25 full-text search with vector cosine similarity, filtered by SQL WHERE clauses, handles this pattern well. The Joern project's **Code Property Graph** (CPG) — merging AST, CFG, and program dependence graphs into a single property graph — provides the gold standard for representing binary structure, and its schema layers (base → AST → CFG → PDG → call graph → findings → tags) map directly to the progressive analysis stages that multi-agent systems need.

## Progressive disclosure through hierarchical summarization

The hardest problem isn't storage — it's **deciding what context a fresh agent needs**. Dumping even a moderately annotated knowledge base into a prompt will overwhelm the context window and degrade agent performance. Research consistently shows that **agent effectiveness decreases with excessive context**, even when the window technically accommodates it.

The most effective approach combines **three layers of progressive disclosure**:

**Layer 1: Hierarchical summaries built bottom-up.** When an agent completes analysis of a function, it generates a structured summary (1–3 sentences of behavior, key observations, confidence level). A periodic background process aggregates function summaries into **module-level summaries**, and module summaries into a **binary-level executive summary**. This creates a tree of summaries at increasing abstraction. A fresh agent receives the binary-level summary (~500 tokens) plus module-level summaries for relevant components (~200 tokens each), and retrieves function-level detail only on demand. This mirrors how human analysts work: start with the big picture, zoom into areas of interest.

**Layer 2: RAG over the knowledge base.** When an agent needs to understand a specific function or investigate a lead, it queries the knowledge base semantically rather than loading everything. Embedding models matter here: **generic text embeddings underperform for code**. The best results come from code-specialized models — CodeSage, Qwen3-Embedding (which tops the MTEB-Code benchmark), or fine-tuned models like BinBcla that treat assembly instructions as tokens. Embed both the decompiled pseudocode and the natural-language summary separately, since agents sometimes search by code structure and sometimes by behavioral description.

**Layer 3: Structured context templates.** Inspired by Birgitta Böckeler's context engineering framework (published February 2026 on Martin Fowler's site), define **always-on context** (binary metadata, architecture, key globals — analogous to CLAUDE.md files), **scope-triggered context** (loaded when the agent's task involves a particular module), and **lazy-loaded skills** (the agent decides when to pull in detailed calling convention docs, crypto algorithm references, etc.). The SWE-Adept coding agent demonstrates a relevant pattern: **agent-directed depth-first search** over dependencies, where the agent selectively traverses the call graph, pulling in only the functions it determines are relevant rather than receiving a pre-selected context dump.

## Off-the-shelf tools form a viable but incomplete stack

The **MCP ecosystem has become the de facto integration layer** between LLMs and RE tools. Every major disassembler now has an MCP server: GhidraMCP exposes **193 tools** via HTTP/REST, ida-pro-mcp provides isolated contexts per agent (the `--isolated-contexts` flag explicitly prevents cross-agent interference), and radare2-mcp/CutterMCP cover the open-source side. These MCP servers let agents query decompiled output, rename functions, set comments, and traverse cross-references without parsing tool-specific formats.

For the underlying RE engine, the choice depends on constraints:

- **Ghidra + PyGhidra + GhidraMCP** is the strongest open-source stack. PyGhidra (now officially integrated) embeds the full Ghidra analysis engine as a Python library via JPype, enabling headless analysis. The Ghidra Server provides check-in/check-out collaboration, though its pessimistic locking model wasn't designed for high-frequency concurrent agent writes — you'd use it for persisting snapshots rather than as the primary shared state.
- **IDA Pro + idalib** offers superior analysis quality and the only built-in multi-agent safety mechanism (`--isolated-contexts`), but requires commercial licensing.
- **BinSync** is uniquely valuable as a cross-decompiler collaboration layer with a normalized artifact schema (REAs — Reverse Engineering Artifacts) stored in Git. It already supports a "ChatGPT User" that participates as an AI collaborator, and its `libbs` library enables programmatic artifact manipulation across IDA, Binary Ninja, and Ghidra.

For the **agent orchestration layer**, the frameworks offer genuinely different strengths:

- **LangGraph** is the most production-ready choice. Its TypedDict state model with reducers handles concurrent agent updates deterministically. Checkpointing to PostgreSQL enables fault-tolerant long-running analysis. The graph-based workflow definition naturally expresses RE analysis patterns with cycles (iterative deepening, re-analysis after new discoveries).
- **Letta (MemGPT)** provides the closest analog to the tiered memory an RE system needs: core memory (current analysis focus) maps to the function under examination, archival memory maps to the full knowledge base. Its shared memory blocks let multiple agents coordinate through a common "project status" block. The V1 architecture's native model tool calling (rather than prompt-injected memory management) reduces overhead.
- **CrewAI** offers the quickest path to a working prototype via its role-based metaphor (triage agent, static analysis agent, vulnerability scanner, report writer), but teams often migrate to LangGraph for production due to CrewAI's limited checkpointing and coarse error handling.

**No single framework solves the whole problem.** The practical approach is: LangGraph or Letta for orchestration and memory management, PostgreSQL + pgvector as the shared knowledge base, MCP servers as the interface to RE tools, and custom code for hierarchical summarization and progressive context assembly.

## What the cutting edge looks like in 2025–2026

The most directly relevant deployed system is **SentinelOne's Adversarial Consensus Engine** (published March 2026). It runs a serial consensus pipeline where specialized agents operate different RE tools (radare2, Ghidra, Binary Ninja, IDA Pro) against the same binary in two rounds. Round 1 extracts findings into a shared context document. Round 2 — called "The Gauntlet" — implements adversarial peer review: each agent must formally **agree or disagree** with every previous claim, with rejected findings tracked alongside rationale. Key design decisions: they use deterministic bridge scripts rather than interactive MCP queries (avoiding 15–50 MCP round-trips per function), and the shared context is carried in-memory through the LLM context window rather than persisted to disk. This works for malware triage but wouldn't scale to large binaries with thousands of functions.

On the academic side, **LmPa** (Purdue, updated December 2024) pioneered the pattern most relevant to multi-agent RE: it divides variable name recovery into many small LLM queries, then uses **program analysis to propagate results across functions** through the call graph — injecting findings as code comments that provide context for subsequent queries. Its successor **GenNm** (NDSS 2025) extends this with preference optimization, improving name recovery accuracy by 5.6–11.4 percentage points. This "many small queries + inter-procedural propagation" pattern is exactly how a multi-agent system should work.

**ReCopilot** (QI-ANXIN XingTu Lab, May 2025) represents the state of the art in multi-task binary analysis LLMs, supporting decompilation, function naming, variable typing, struct recovery, and summarization from a single **7B parameter model** fine-tuned on Qwen2.5-Coder. It uses data flow analysis and call graph context to enhance prompts, and its authors explicitly identify the next step as building an **agentic mode** for autonomous multi-step analysis. The "Beyond C/C++" paper (June 2025) proposes a three-layer architecture — Program Analysis Engine producing facts with confidence levels, Probabilistic Inference Engine aggregating them, LLM providing context-aware reasoning — that maps naturally to a multi-agent blackboard system.

The **blackboard architecture** has been validated for LLM multi-agent systems by the LbMAS paper (Han & Zhang, July 2025), which demonstrated competitive performance with state-of-the-art on standard benchmarks while spending fewer tokens. Its three components — shared blackboard, dynamic control unit, and specialized agent pool (including a Decider, Planner, Critic, and Conflict-Resolver) — are directly applicable to RE. The blackboard serves as the knowledge base; the control unit selects which agent (string analyst, control flow expert, crypto specialist) should act next based on the current analysis state; the Conflict-Resolver handles contradictory hypotheses about function behavior.

## A concrete architecture for the system

The recommended architecture layers five components:

**1. Analysis Engine Layer.** Ghidra running headless via PyGhidra performs initial auto-analysis, producing the baseline function list, disassembly, preliminary decompilation, cross-references, string references, and type recovery. This populates the knowledge base with the structural skeleton. Multiple Ghidra instances can analyze different sections in parallel.

**2. Knowledge Base Layer.** PostgreSQL + pgvector stores everything. The schema described above handles structured metadata, embeddings, and annotations. A **write API** enforces schema validation and automatically computes embeddings when agents insert or update function summaries. An **event bus** (PostgreSQL LISTEN/NOTIFY or a lightweight message queue) notifies agents when relevant knowledge base entries change — enabling reactive analysis where discovering a new crypto function triggers the crypto specialist agent.

**3. Context Assembly Layer.** This is the critical middleware between the knowledge base and the agents. It implements three retrieval strategies that agents invoke via tool calls:

- `get_binary_overview()` — returns the executive summary, module list with summaries, key statistics (function count, library ratio, notable strings, import categories)
- `get_module_context(module_id)` — returns the module summary, its constituent functions with one-line descriptions, internal call graph, and the most relevant annotations
- `get_function_detail(function_id, depth=1)` — returns full decompiled code, all annotations, immediate callers/callees (with their summaries), referenced strings and data, and similar functions found via embedding search
- `search_knowledge(query, filters)` — hybrid semantic + keyword + structural search with optional filters (module, tag, confidence threshold, annotating agent)

Each retrieval method produces a **context budget-aware** response, truncating or summarizing content to fit within a specified token limit. The context assembly layer also maintains the **hierarchical summary tree**, rebuilding module and binary summaries incrementally as function-level knowledge accumulates.

**4. Agent Orchestration Layer.** LangGraph manages agent workflows with a shared TypedDict state containing the current analysis objectives, assigned regions, and coordination metadata. The orchestrator agent operates at the binary level, identifying unexplored regions and spawning specialist agents. Specialist agents include: a **triage agent** (rapid function classification — library, thunk, interesting, crypto, network, string manipulation), a **deep analysis agent** (thorough single-function reverse engineering), a **cross-reference agent** (tracing data flow and call chains across functions), and a **synthesis agent** (building module-level understanding from constituent function analyses). Each agent reads from and writes to the knowledge base, never communicating directly with other agents — pure blackboard pattern.

**5. Annotation Consensus Layer.** When multiple agents analyze the same function, their annotations may conflict. A lightweight consensus mechanism resolves conflicts: higher-confidence annotations take precedence; when confidence is similar, the most recent analysis wins; the synthesis agent periodically reviews conflicting annotations and produces authoritative summaries. All superseded annotations remain in the version chain for auditability.

## Practical tradeoffs and what to build first

**Start with PostgreSQL + pgvector, not a multi-database architecture.** The synchronization overhead of maintaining Neo4j alongside PostgreSQL is significant, and PostgreSQL's recursive CTEs handle call graph queries for binaries under ~50K functions. Add Neo4j only if you find yourself needing complex graph algorithms (community detection, graph neural network features) that PostgreSQL can't express efficiently.

**Use BinExport as your initial data loader.** Google's BinExport protobuf format captures disassembly, basic blocks, CFG, call graph, and comments from IDA Pro, Ghidra, or Binary Ninja into a compact, well-documented format. Write an importer from BinExport into your PostgreSQL schema, and you immediately support all three major RE tools as analysis frontends.

**Embedding model selection matters more than vector database selection.** For function-level code embeddings, fine-tuned models like **Qwen3-Embedding** or CodeSage substantially outperform generic sentence-transformers. For natural-language summaries and annotations, standard models like `all-MiniLM-L6-v2` work well. Maintain separate embedding columns for code and summaries — agents search differently depending on whether they're looking for structural similarity or behavioral similarity.

**The biggest unsolved problem is automatic module identification.** Grouping hundreds of functions into coherent logical modules (the crypto subsystem, the network stack, the command parser) is crucial for hierarchical summarization but difficult to automate. Graph community detection on the call graph provides a starting point, but the best results will come from an agent that combines structural clustering with semantic analysis of function summaries. This is where the graph database tier adds the most value — algorithms like Louvain community detection on call graph edges, weighted by cross-reference density, can propose initial module boundaries that a synthesis agent then refines.

## Conclusion

The multi-agent RE shared state problem sits at the intersection of three maturing fields: LLM-assisted binary analysis (where specialized 7B models now rival GPT-4o on decompilation), agent orchestration frameworks (where LangGraph and Letta provide production-grade state management), and collaborative RE tooling (where MCP servers have unified the interface between LLMs and disassemblers). The key architectural insight is that **the knowledge base should implement the blackboard pattern** — agents communicate exclusively through shared state, never directly — with **progressive disclosure via hierarchical summarization** replacing the temptation to dump everything into context. PostgreSQL + pgvector provides the most pragmatic foundation: a single ACID-compliant store supporting relational queries, graph traversal, vector similarity search, and full-text search, with a schema that captures both the structural skeleton of a binary and the evolving understanding that agents build collaboratively. The tools exist today to build this system by composing Ghidra/PyGhidra for analysis, BinExport for data interchange, LangGraph for orchestration, and a custom context assembly layer that gives each agent exactly the context it needs — no more, no less.
